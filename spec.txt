# VPN v1 Specification

## 0. Scope

This spec defines a VPN that:

* multiplexes public traffic and VPN traffic on the same public endpoints (`:80` TCP, `:443` TCP/UDP),
* forwards unauthenticated traffic to a local web service (nginx) on internal ports,
* uses a TUN device (L3) on client and server,
* reuses the security of TCP/TLS and a QUIC-shaped UDP packet protection layer (custom; not interoperable QUIC),
* requires explicit client authentication at the VPN protocol layer before any tunneled data is accepted,
* prefers UDP-QSP for data, with TCP fallback, and performs a UDP-QSP “verify” step before switching active transport.

## 1. Terminology

* **Wrapper**: the front process that owns public sockets and routes traffic to either VPN handlers or nginx.
* **VPN server**: the VPN handler(s) that terminate TLS for claimed clients and perform VPN authentication + TUN forwarding.
* **NAT forwarder path**: wrapper path that forwards UDP/443 to nginx (internal QUIC/H3).
* **Claimed**: traffic identified by the wrapper as intended for the VPN (based on a deterministic predicate).
* **Unknown**: traffic not claimed; TCP is passed through to nginx; UDP is forwarded to nginx per policy below.
* **Active transport**: the currently used channel for data for a given client (`TCP` or `UDP-QSP`).
* **UDP-QSP**: custom QUIC-shaped packet format used after registration (short headers only).

## 2. Network Topology

### 2.1 Public ports

* `:80/tcp`  → nginx `127.0.0.1:80/tcp` (HTTP, ACME, redirect to HTTPS)
* `:443/tcp` → wrapper owns (routes to VPN TCP handler or TCP passthrough to nginx TLS)
* `:443/udp` → wrapper owns (routes to VPN UDP-QSP handler or UDP forwarder to nginx QUIC)

### 2.2 Internal ports

* nginx listens on:

  * `127.0.0.1:8443/tcp` (TLS + HTTP/1.1 and/or HTTP/2)
  * `127.0.0.1:8443/udp` (HTTP/3 / QUIC)
* Internal ports are firewall-blocked from the internet.

### 2.3 Alt-Svc

* nginx should advertise `Alt-Svc: h3=":443"` in HTTPS responses.
* Even though nginx listens on `8443`, the wrapper is responsible for exposing UDP/443 externally.

## 3. Wrapper Responsibilities

### 3.1 Ownership

* Wrapper is the **only** process that reads from public `:443/udp`.
* Wrapper is also the owner of public `:443/tcp` accept loop.
* Wrapper must ensure that traffic is routed exclusively:

  * claimed traffic goes only to VPN handlers,
  * unknown traffic goes only to nginx forwarder/passthrough.

### 3.2 Classification

Wrapper must implement:

* `classify_tcp_client_hello(bytes) -> Verdict`
* `classify_quic_datagram(bytes) -> Verdict`

`Verdict`:

* `CLAIM`: route to VPN handler
* `PASS`: route to nginx path
* `DROP`: drop silently

**Hard requirements for classifier**:

* bounded CPU per packet/connection (no unbounded parsing),
* bounded memory (no allocations proportional to attacker input),
* deterministic behavior for a given input,
* does not emit any network packets itself.

#### 3.2.1 TCP classification

For TCP connections, the client includes a token in the TLS ClientHello `legacy_session_id` field:

```
session_id = part1 || part2

part1 = MAC(random[0:N] || server_secret)
part2 = MAC(key_share || server_secret)
```

Parameters (fixed):
* `legacy_session_id` length is **32 bytes**.
* `MAC` is HMAC-SHA256, truncated to 16 bytes.
* `N = 16` (bytes from the ClientHello `random` field).
* `part1_len = 16`, `part2_len = 16`.
* `key_share` is **x25519 only** (32-byte key share).

Where `random` is the ClientHello's 32-byte random field (first N bytes used), and `key_share` is extracted from the `key_share` extension.

Classification logic:
1. Extract `legacy_session_id` from ClientHello
2. If `legacy_session_id` length != 32 → `PASS` (early return)
3. Parse and verify `part1` using `random[0:N]`
4. If `part1` invalid → `PASS` (early return)
5. Parse `key_share` extension and verify `part2`
6. If `key_share` missing, not x25519, or not 32 bytes → `PASS`
7. If `part2` invalid → `PASS`
8. Both valid → `CLAIM`

Failed claims route to nginx normally, making VPN and nginx traffic indistinguishable to observers.

#### 3.2.2 UDP classification (QUIC-shaped)

For UDP/443, the wrapper does **not** attempt to claim based on QUIC Initial packets. All long headers are forwarded to nginx.
Rationale: extracting TLS key_share from QUIC Initial requires CRYPTO frame reassembly/reordering and deeper parsing; by keeping classification TCP-only, the wrapper remains simple and bounded.

Classification logic:
1. If datagram is not recognized as QUIC-format → `DROP`
2. If packet header form is **long** → `PASS` (forward to nginx)
3. If packet header form is **short**:
   * If DCID exists in wrapper's `cid_map` → `CLAIM`
   * Otherwise → `PASS`

### 3.3 TCP routing behavior (`:443/tcp`)

* Wrapper reads enough bytes to extract ClientHello (or a bounded prefix sufficient for `classify_tcp_client_hello`).
* If `PASS`:

  * wrapper opens `127.0.0.1:8443/tcp` and pipes bytes bidirectionally (pure passthrough; wrapper does not terminate TLS).
* If `CLAIM`:

  * wrapper hands the connection to VPN TCP handler which **terminates TLS** and runs VPN protocol.
* If `DROP`:

  * wrapper closes the TCP connection.

### 3.4 UDP routing behavior (`:443/udp`)

Wrapper receives each datagram and applies policy:

1. **QUIC format check**: If datagram is not recognized as QUIC → `DROP`.
2. **Parse QUIC header**:
   * **Long header**:
     * Always forward to nginx via NAT (section 3.5).
   * **Short header**:
     * If DCID exists in wrapper's `cid_map` → route to VPN UDP-QSP handler.
     * Otherwise → forward to nginx via NAT (section 3.5).

This design keeps real QUIC handshakes on the wire (client↔nginx) while allowing VPN traffic to use QUIC-shaped short headers after DCID registration.

### 3.5 UDP forwarder to nginx (4-tuple NAT)

For non-clients, the wrapper implements a 4-tuple NAT:

* **NAT map**: `client_ip:client_port -> local_port`
  * `client_ip:client_port`: source address from incoming packet
  * `local_port`: port of wrapper's socket that forwards to nginx
* **Forwarding**: Each mapped client gets its own UDP socket to `127.0.0.1:8443/udp` (nginx).
* **Behavior**:
  * DCID changes are invisible to wrapper (same 4-tuple → same socket → nginx handles QUIC state).
  * Source changes (real connection migration) create new socket → nginx sees new connection and issues PATH_CHALLENGE per QUIC spec.
* **Resource limits**:
  * NAT table is bounded (configurable, e.g., 100-1000 entries).
  * Oldest entries evicted (LRU) when limit is reached.
  * Idle entries expire after timeout.

### 3.6 CID Map

* Wrapper owns `cid_map: CID -> ConnHandle` for VPN UDP-QSP sessions.
* The map is updated only via the authenticated TCP control channel:

  * `AddCid { cid, conn_handle, keys, pn_start, key_phase, ack }`
  * `RemoveCid { cid }`
* Wrapper must acknowledge `AddCid` before the client is allowed to send UDP-QSP packets with that CID.
* `keys` contains:

  * `hp_tx`, `hp_rx` (header protection keys),
  * `aead_tx`, `aead_rx` (payload keys),
  * `iv_tx`, `iv_rx` (payload IVs),
  * `cipher` (AES-128-GCM or ChaCha20-Poly1305).

## 4. Identity, Config, Revocation

### 4.1 Static config

* Server config is static. To add/remove clients:

  * edit config,
  * restart server.
* Revocation = remove client entry (or mark disabled) + restart.

### 4.2 Client identity

Each client has:

* `client_id` (16 bytes)
* `pubkey_ed25519` (32 bytes)
* `assigned_ipv4` (static VPN IP)

Client config stores:

* its `client_id`,
* its `assigned_ipv4`,
* its private key for authentication.

Server config stores:

* list of clients with `client_id`, `pubkey`, `assigned_ipv4`.

### 4.3 Binding identity to IP (authorization)

During VPN authentication:

* client presents `client_id` and a proof (signature) over a server-provided challenge and context.
* server verifies:

  * `client_id` exists and is enabled,
  * signature is valid for configured pubkey,
  * session is bound to `assigned_ipv4` from config.

### 4.4 Packet spoofing prevention (mandatory)

For every tunneled packet received from a client session:

* parse inner IPv4 header and enforce: `src_ip == session.assigned_ipv4`.
* drop packets that violate this.

## 5. Transport Security

### 5.1 TCP

* VPN TCP handler terminates TLS and runs VPN protocol over the TLS stream.
* Unknown TCP connections are passed through to nginx without termination.

TLS library: **BoringSSL** via `boring-sys` (patched client hook to set `legacy_session_id` after `random` and `key_share` are ready, before ClientHello transcript hashing).

### 5.2 UDP-QSP (QUIC-shaped, custom)

* UDP-QSP uses QUIC short headers and packet protection for **wire-shape only**.
* There is **no** QUIC handshake on the VPN path; all QUIC long-header traffic is forwarded to nginx.
* Keys are negotiated over the authenticated TCP control channel and are **not** QUIC/TLS keys.
* Required properties:

  * Short header format with fixed bit = 1.
  * QUIC-style header protection (HP) to mask PN length / key phase / reserved bits.
  * QUIC-style AEAD payload protection with nonce = `IV XOR PN` and associated data = unmasked header.
  * Payload length sufficient for the 16-byte HP sample (pad if needed).
* Interop: UDP-QSP is **not** compatible with standard QUIC stacks.

Client-side QUIC library (wire-shape handshake only): **quiche** (vendored + patched as needed).

Constraints / patch notes for `quiche`:
* Must allow controlling Initial DCID length to match Chrome.
* Must allow including unknown transport parameters (or an escape hatch to append raw TP bytes).
* Must not require owning a socket (quiche is acceptable; quinn is not due to `SocketAddr` requirements and `ring` dependency).

## 6. VPN Protocol (Application Layer)

### 6.1 Common rules

* No `DATA` is accepted or forwarded until authentication completes.
* Authentication is replay-resistant via server challenge.
* After authentication, session is marked `AUTHENTICATED` and data plane is enabled.

### 6.2 Message types

A minimal set:

* `AUTH` (client → server)
* `AUTH_OK` / `AUTH_FAIL` (server → client)
* `REGISTER_CID` / `REGISTER_OK` / `REGISTER_FAIL` (client → server / server → client)
* `PING` / `PONG` (both)
* `CLOSE` (both)
* `DATA` (both, carries raw IP packet bytes)

### 6.3 TCP framing

Over the TLS stream:

```
TYPE: u8
LEN:  u32 (big-endian)
PAYLOAD: LEN bytes
```

Constraints:

* `LEN` must be bounded by config.
* Before auth, only `AUTH`, `PING`, `CLOSE` allowed.
* `REGISTER_CID` is only allowed after `AUTH_OK`.
* `DATA` payload must not exceed `tun_mtu + headroom` (recommend: `tun_mtu + 64`).

### 6.4 UDP-QSP mapping

* Control remains on the TCP channel.
* Data and keepalive over UDP-QSP use a single framed message per UDP datagram:

  * `TYPE: u8`
  * `LEN: u32 (big-endian)`
  * `PAYLOAD: LEN bytes`
* `DATA` payload is exactly one IP packet.
* `PING`/`PONG` may be sent over UDP-QSP for liveness.

## 7. Keepalive, Timeouts, Switching

### 7.1 Keepalive

* Send periodic `PING` on the **active transport** only.
* Ping interval is randomized per connection:

  * `ping_interval = uniform(ping_min, ping_max)`

### 7.2 Timeouts

Configurable timeouts:

* `auth_timeout`: maximum time from connection accepted to authenticated state.
* `idle_timeout`: no inbound traffic on active transport triggers disconnect.
* `udp_verify_timeout`: time allowed for UDP verify step (below).

### 7.3 Transport preference and switching

* Startup: client connects via TCP, authenticates, starts data.
* Wire-shape step: client establishes real QUIC to nginx (client-side QUIC library).
* The nginx QUIC connection is not used for VPN data; it is only for on-wire handshake and DCID reuse.
* Registration: client sends `REGISTER_CID` over TCP with:

  * DCID to reuse on UDP-QSP,
  * UDP-QSP keys (hp + payload, both directions),
  * initial PN and key phase.
* Verify-before-switch:

  1. server sends `PING` over UDP-QSP
  2. client replies `PONG` within `udp_verify_timeout`
  3. only then server switches `active_transport := UDP-QSP`
  4. server keeps TCP session idle as fallback, but data uses only active

Session limits:

* At most **one active session total** per client at a time.
* A newly verified UDP-QSP session replaces TCP as active.
* If active UDP-QSP fails (idle timeout / loss of keepalive):

  * fall back to TCP if TCP is still connected and authenticated; otherwise client reconnects.

## 8. TUN Data Plane

### 8.1 Client

* External script:

  * creates TUN interface,
  * assigns `assigned_ipv4/32`,
  * installs routes as desired (full-tunnel or split-tunnel),
  * configures DNS if desired.
* VPN client process:

  * reads IP packets from TUN and sends them via active transport as `DATA`,
  * receives `DATA` and writes packets to TUN.

### 8.2 Server (Model A: OS handles forwarding/NAT)

* External script:

  * creates server TUN interface,
  * assigns server VPN IP (gateway IP),
  * enables forwarding,
  * configures NAT masquerade for VPN subnet to WAN.
* VPN server process:

  * maintains a map of `assigned_ipv4 -> client_session` for all connected clients.
  * on receiving client `DATA`, writes the IP packet into the server TUN device.
  * on reading from TUN:

    * if destination IP matches a connected client's `assigned_ipv4`, send to that client as `DATA`,
    * otherwise drop (such packets should not appear on TUN under correct routing; the kernel would route non-VPN traffic directly out the WAN interface).

### 8.3 MTU

* Default `tun_mtu = 1280` (safe baseline).
* Configurable on both client and server.
* Scripts must apply MTU:

  * `ip link set dev <tun> mtu <tun_mtu>`

## 9. OS Setup (Server)

### 9.1 Sysctls

Minimum:

* `net.ipv4.ip_forward=1`

Optional if you later enable:

* IPv6 forwarding equivalents.

### 9.2 NAT (example nftables)

Example concept (interface names and subnet configurable):

* VPN subnet: `10.10.0.0/24`
* WAN interface: `eth0`

nftables concept:

* masquerade `ip saddr 10.10.0.0/24 oifname "eth0"`

(Exact rules are environment-specific; the spec requires equivalent behavior.)

## 10. External Script Contract (WireGuard-like)

### 10.1 Server script inputs

* VPN subnet CIDR
* server VPN IP
* TUN name (default `tun0`)
* WAN interface name
* `tun_mtu`

### 10.2 Client script inputs

* assigned VPN IP
* TUN name (default `tun0`)
* `tun_mtu`
* route directives (user-managed)

### 10.3 App expectations

* App fails fast if TUN device is missing/unusable.
* App does not attempt to modify routes.

## 11. Configuration Files (Recommended Schema)

### 11.1 Server config (TOML-style)

* `listen_tcp = "0.0.0.0:443"`
* `listen_udp = "0.0.0.0:443"`
* `nginx_tcp_upstream = "127.0.0.1:8443"`
* `nginx_udp_upstream = "127.0.0.1:8443"`
* `tun_name = "tun0"`
* `tun_mtu = 1280`
* timeouts/keepalive:

* `ping_min_ms`, `ping_max_ms`, `auth_timeout_ms`, `idle_timeout_ms`, `udp_verify_timeout_ms`
* clients list:

  * `client_id`, `pubkey_ed25519`, `assigned_ipv4`

### 11.2 Client config

* server address
* `client_id`, `assigned_ipv4`
* `privkey_ed25519`
* `tun_name`, `tun_mtu`
* optional: preferences for upgrade/fallback timing

## 12. Observability

* Log events:

  * connection accepted (tcp/udp),
  * claimed vs passed vs dropped,
  * auth success/fail with client_id,
  * session switch TCP↔UDP-QSP,
  * disconnect reasons (timeout, close, restart).
* Counters:

  * packets forwarded to nginx (udp/tcp),
  * packets routed to VPN,
  * auth failures,
  * drops due to bounds/timeouts.
